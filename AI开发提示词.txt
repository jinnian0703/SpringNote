二、开发总目标

实现一个名为 SpringNote 的 Windows 桌面应用。

SpringNote 是给实习生用的 AI 智能便签，核心功能是：

* 首页随手记录工作内容
* AI 将随手记录结构化为完成事项、问题记录、明日计划
* 自动合并写入当日日报 Markdown
* 生成日报、周报、月报
* 便签页编辑 Markdown 源码并实时预览
* 便签页支持 FIM 编辑补全
* 回忆书支持基于历史记录的 AI 问答
* 设置页支持偏好设置、供应商、默认模型、快捷键、统计、关于
* 桌面组件支持工作计时、金币收益、左键暂停/开始、右键打开首页

不要做成普通聊天软件，也不要做成任务管理器。它是 AI 驱动的工作记录、日报生成和 Markdown 编辑工具。


三、工程原则

1. Flutter 负责 UI 和交互。

2. Rust 负责本地核心能力，至少包括：

* 文件系统读写
* config.json 读写
* Markdown 文件管理
* SQLite 统计数据读写
* AI 模型调用
* 三种 AI 协议接口适配
* 必要的 Windows 本地能力封装

3. AI HTTP 请求、供应商协议适配、Token 统计、调用日志、错误归一化必须放在 Rust 层实现。Flutter 不直接请求 AI API。

4. Flutter 与 Rust 通过 flutter_rust_bridge 通信。

5. 不要把所有逻辑堆到 main.dart。

6. 不要直接把 HTML 原型塞进 WebView。HTML 只是视觉参考，真实应用应使用 Flutter Widget 实现。

7. UI 不要使用 Flutter 默认紫色 Material 风格。必须贴近原型中的极简、浅色、Slate 灰阶、Apple / Linear / OpenAI 风格。

8. 先实现可运行 MVP，再逐步完善 AI、统计、组件和复杂交互。

9. 每次实现一个阶段后必须运行检查：

* flutter analyze
* flutter test
* cargo fmt
* cargo test

如果某项暂时无法运行，必须说明原因，并保证代码结构不阻塞后续开发。

八、AI 调用规则

AI 调用必须在 Rust 层实现。

Flutter 只能调用 Rust 暴露的方法，例如：

* generate_structured_note
* merge_daily_note
* generate_weekly_report
* generate_monthly_report
* fim_complete
* memory_chat
* test_provider_connection
* fetch_provider_models

Flutter 不允许直接保存 API Key 后自己发 HTTP 请求。

Rust 侧需要实现统一 AI service，并在内部适配三种协议接口：

1. OpenAI 兼容协议

用于 OpenAI、OpenRouter、本地 OpenAI-compatible 服务、DeepSeek、Qwen 兼容服务等。

配置字段：

* baseUrl
* apiPath
* apiKey
* modelId

调用方向：

* 聊天：/chat/completions 或用户配置的 apiPath
* 补全 / FIM：/completions 或供应商配置的补全路径

2. Google Gemini 协议

用于 Google / Gemini 供应商。

Rust 侧需要把统一请求转换为 Gemini generateContent / streamGenerateContent 风格。

配置字段：

* baseUrl
* apiKey
* modelId

modelId 用于拼接 Gemini 模型路径或请求参数。

3. Anthropic Claude 协议

用于 Claude 供应商。

Rust 侧需要把统一请求转换为 Anthropic Messages API 风格。

配置字段：

* baseUrl
* apiKey
* modelId

请求需要由 Rust 侧补齐 Anthropic 需要的 Header，例如版本 Header。不要在 Flutter 层处理这些协议细节。

Rust 侧统一 AI service 需要对外屏蔽协议差异。Flutter 只传业务意图、模型 ID、上下文和用户输入，不关心底层协议。

供应商配置字段：

* enabled
* name
* protocol
* apiKey
* baseUrl
* apiPath
* models

不要在添加供应商弹窗中实现 Use Responses API。

AI 客户端需要支持：

* 三种协议接口：OpenAI 兼容、Google Gemini、Anthropic Claude
* chat 类调用
* FIM 补全调用
* 历史记录问答调用
* 供应商连接测试
* 模型列表获取
* token 用量记录
* 错误处理

Token 统计、模型调用记录、错误记录应在 Rust 层写入 SQLite。

如果某协议暂时无法精确获取 Token 用量，应在 Rust 层做兼容处理，允许记录 0 或估算值，但不能让 UI 崩溃。

如果某供应商接口不可用，UI 应显示低打扰错误，不要崩溃。

首页输入提交流程：

1. 用户输入原始工作记录。
2. 调用智能生成模型。
3. 输出结构化内容。
4. 更新首页今日工作概览。
5. 读取当日日报 Markdown。
6. 将新结构化内容和旧日报交给 AI 合并整理。
7. 写回 notes/daily/YYYY-MM-DD.md。
8. 更新统计：智能生成次数、token、热力图活跃度。

如果未配置模型，先使用本地 mock 结构化结果，保证应用可运行。

十六、开发过程要求

每次开发前：

1. 先阅读相关文档。
2. 检查当前代码结构。
3. 明确本次实现范围。

每次开发中：

1. 不要一次性大改所有模块。
2. 优先保证可运行。
3. 数据模型和服务层先稳定，再做复杂 UI。
4. 不要引入无关依赖。

每次开发后：

1. 运行 dart format。
2. 运行 flutter analyze。
3. 运行 flutter test。
